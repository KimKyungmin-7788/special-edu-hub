# PRD — 강원 특수교육 디지털 학습자료 누리집 (가칭)

> **문서 목적**
> 이 문서는 Claude Code가 개발을 시작하기 전에 읽는 제품 요구사항 문서(PRD)다.
> 이번 범위는 **0~1단계 "작동하는 카탈로그"** 까지이며, 회원/인증/등록/소셜 기능은 다음 단계로 미룬다.
> 디자인은 의도적으로 **중립(화이트·블랙·그레이)** 으로 시작하고, 브랜드는 나중에 토큰 교체로 입힌다.

---

## 1. 한 줄 요약

특수교육 교사들이 직접 만든 디지털 교육자료(웹앱)를 **카테고리별로 모아 보여주는** 공개 카탈로그 누리집. 이번 단계의 목표는 **방문자가 앱을 둘러보고, 상세 정보를 확인하고, 새 탭에서 앱을 실행**할 수 있는 상태까지다.

---

## 2. 범위 (Scope)

### 이번에 만드는 것 (In Scope)

**0단계 — 골격**
- 중립 디자인 토큰(CSS 변수) 세팅
- 전체 레이아웃 골격(헤더/네비/푸터)
- 페이지 라우팅
- 백엔드 없이 **샘플 데이터(목업)** 로 화면 모양 완성

**1단계 — 카탈로그 작동**
- Supabase 연결
- 앱 목록(카탈로그) — **읽기 전용**
- 카테고리(과목·업무) 분류 및 필터
- 앱 상세 페이지 — **읽기 전용**
- 시드 데이터(실제 앱 몇 개)로 진짜처럼 보이게

### 이번에 만들지 않는 것 (Out of Scope — 다음 단계)

아래는 **UI 자리(껍데기)만** 두거나 "준비 중"으로 표시한다. 실제 작동은 후속 단계.

- 회원가입 / 로그인 (2단계)
- 교사인증 신청·승인 (2단계)
- 앱 등록 기능 (3단계)
- 좋아요 / 담기 **실제 누적·토글** (3단계) — *카드의 숫자는 시드 데이터로 표시만*
- 조회수 **실제 누적** (3단계) — *상세 페이지에 숫자 표시만*
- 댓글 / 답글 / 멘션 (4단계)
- 브랜드 / 본 디자인 적용 (5단계)

> **중요:** 위 기능들의 버튼·영역이 화면에 보일 수 있으나, 이번 단계에서는 동작하지 않거나 비활성 상태다. Claude Code는 이번 범위 밖 기능을 임의로 구현하지 않는다.

## 3. 기술 스택

| 영역 | 선택 |
|---|---|
| 프론트엔드 | React + Vite |
| 스타일 | Tailwind CSS + shadcn/ui |
| 백엔드/DB | Supabase (1단계부터) |
| 배포 | Vercel (추후) |

선정 이유: 사용자가 이미 보유·숙련한 스택이며, shadcn/ui가 색·폰트·반경을 **CSS 변수 토큰**으로 관리해 "디자인 나중 교체" 전략과 정확히 맞는다.

---

## 4. 디자인 원칙 (가장 중요)

1. **중립으로 시작.** 화이트 / 블랙 / 그레이 + 기본 시스템 폰트. 미완성처럼 보이지 않는 "담백한 미니멀"을 목표로 한다.
2. **색·폰트·반경·여백은 전부 CSS 변수 토큰으로만 사용.** 컴포넌트 안에 색을 직접 박지 않는다(하드코딩 금지). → 브랜드 확정 시 토큰 파일 한 곳만 바꾸면 전체가 바뀐다.
3. **브랜드 요소(로고·누리집명·히어로 문구)는 config로 분리.** 화면 곳곳에 직접 쓰지 않고 한 곳에서 불러온다.
4. **화면 구조(레이아웃·정보 배치·페이지 흐름)는 지금 제대로 잡는다.** 나중에 바꾸는 건 "표면 스타일"이지 "구조"가 아니다.

## 5. 사이트 구조 & 페이지 명세

### 5.1 공통 레이아웃

- **헤더(상단 고정)**
  - 좌측: 로고 자리(미정 — 임시 텍스트 로고)
  - 중앙/우측: 네비게이션 바
  - 우측 끝: 로그인 / 회원가입 버튼 *(이번 단계는 모양만, 비작동)*
- **푸터**: 누리집명, 간단한 안내 문구

### 5.2 랜딩 페이지 (참고: 레퍼런스 이미지 1 "에듀베이지")

위에서 아래로 다음 순서:

1. **히어로 영역**
   - 기본: 문구형 ("교과서만으로는 수업이 힘든 특수교사가 직접 만든 디지털 교육자료")
   - 배너 이미지로 교체 가능한 구조 → **캐러셀(슬라이드)** 로 구현해 문구/배너를 슬라이드로 갈아끼울 수 있게
2. **카테고리별 진입 영역** — 아이콘 + 라벨 그리드 (과목/업무)
3. **최신 앱 목록** — 카드형 그리드

### 5.3 카드 명세 (앱 목록 카드)

카드에 표시할 정보 (확정):
- 썸네일
- 제목
- 카테고리·과목 태그
- 좋아요 / 담기 수 *(시드 데이터로 표시)*

> 조회수는 카드에 넣지 않고 **상세 페이지에만** 둔다.

### 5.4 서브 페이지

| 페이지 | 이번 단계 상태 |
|---|---|
| 전체 | ✅ 작동 (앱 카탈로그 전체) |
| 과목별 | ✅ 작동 (과목 필터 카탈로그) |
| 업무혁신 | ✅ 작동 (업무 카테고리 카탈로그) |
| 수업실천사례 | 🚧 자리만 / "준비 중" |
| 자유게시판 | 🚧 자리만 / "준비 중" |
| 교사인증센터 | 🚧 자리만 / "준비 중" |

### 5.5 앱 상세 페이지 (참고: 레퍼런스 이미지 2 "PBL스케치")

표시 요소:
- 앱 제목 / 개발자(작성자) 이름
- **앱 열기** 버튼 → 새 탭에서 실행 (`target="_blank" rel="noopener"`)
- 공유 버튼 *(이번 단계: 링크 복사 정도만 또는 비활성)*
- 담기 버튼 *(모양만, 비작동)*
- 조회수 표시 *(시드 숫자 표시만)*
- **개발자 소개 글(블로그형)** — 이 앱이 무엇인지 / 개발 이야기. 시드 데이터의 본문 텍스트를 렌더링
- 댓글 영역 *(자리만 / "준비 중" 또는 비작동)*
- 좋아요 버튼 *(모양만, 비작동)*

## 6. 데이터 모델 (1단계 Supabase — 카탈로그에 필요한 최소)

> 이번 단계는 **읽기 전용**. 쓰기(등록/좋아요/댓글)는 후속 단계.

**apps** — 앱(교육자료)
- `id`
- `title` (제목)
- `app_url` (실행 URL, 새 탭 대상)
- `thumbnail_url` (썸네일)
- `author_name` (개발자 표시명) *— 회원 연동 전까지는 텍스트*
- `description` (블로그형 소개 본문, 마크다운 또는 텍스트)
- `view_count` (조회수 — 시드값, 누적은 후속)
- `like_count` (좋아요 수 — 시드값, 누적은 후속)
- `bookmark_count` (담기 수 — 시드값, 누적은 후속)
- `created_at`

**categories** — 분류 (과목·업무)
- `id`
- `name` (예: 국어, 수학, 업무혁신 …)
- `type` ('subject' | 'work' 등)
- `icon` (아이콘 식별자)
- `sort_order`

**app_categories** — 앱↔카테고리 다대다 연결
- `app_id`
- `category_id`

> 카테고리 목록은 **config/시드로 관리**해 나중에 쉽게 추가·수정. 시작 카테고리(특수교육 맥락 반영, 조정 가능): 국어 · 수학 · 사회 · 과학 · 음악 · 미술 · 체육 · 일상생활 · 진로직업 · 창체 · 학급경영 · 업무혁신.

> RLS: 이번 단계는 공개 읽기(SELECT)만 허용. 쓰기 정책은 2~3단계에서 추가.

## 7. 시드 데이터

실제 보유 앱으로 시드를 구성해 1단계에서 진짜처럼 보이게 한다. 예:
- 날짜 저요저요! (달력 학습 앱)
- 멀티보드 (전자칠판 도구)
- 직업가치관 월드컵
- 야옹야옹 고양이 숫자 놀이
- (이하 추가)

각 시드 항목에 제목 / URL / 썸네일 / 카테고리 / 소개 본문 / 임의의 좋아요·담기·조회수 숫자를 채운다.

## 8. 단계별 개발 계획 (이번 범위 내)

| 단계 | 내용 | 백엔드 | 완료 확인 |
|---|---|---|---|
| 0 | 디자인 토큰 + 레이아웃 골격 + 라우팅 + 목업 데이터 | 없음 | 브라우저에서 모든 페이지가 목업으로 표시됨 |
| 1 | Supabase 연결 + 스키마 + 시드 + 카탈로그/상세 읽기 | Supabase | 시드 앱이 목록·상세에 뜨고, 카테고리 필터 작동, 앱 열기 작동 |

- **0→1을 한 번에 시키지 않는다.** 0단계 완료·확인 후 1단계 진행.
- 각 단계 종료 시 Git 커밋.
- Supabase 스키마·RLS SQL은 Claude Code가 작성하되, **사용자가 Supabase 대시보드에서 직접 실행**한다.

## 9. 완료 기준 (Acceptance — 이번 범위)

- [ ] 화이트·블랙·그레이 중립 디자인, 색·폰트가 전부 토큰으로 분리되어 있다.
- [ ] 로고·누리집명·히어로 문구가 config로 분리되어 한 곳에서 수정 가능하다.
- [ ] 랜딩: 히어로(캐러셀) + 카테고리 그리드 + 최신 앱 카드 목록이 보인다.
- [ ] 전체 / 과목별 / 업무혁신 페이지에서 카탈로그가 보이고 카테고리 필터가 작동한다.
- [ ] 앱 상세 페이지에서 제목·개발자·소개 본문·조회수가 보이고, **앱 열기**가 새 탭으로 실행된다.
- [ ] 수업실천사례 / 자유게시판 / 교사인증센터는 "준비 중" 자리로 존재한다.
- [ ] 범위 밖 기능(로그인/등록/좋아요 토글/댓글)은 동작하지 않으며, 모양만 있거나 준비 중이다.
- [ ] 데이터가 Supabase 시드에서 불러와진다(1단계).

## 10. 비고

- 앱은 모두 **새 탭(링크) 방식**으로 실행한다(iframe 아님). 각 앱이 화면을 꽉 채워 동작하고, 누리집과 앱이 완전히 독립된다.
- 브랜드/본 디자인은 5단계에서 토큰 파일·로고 교체로 적용한다. 이번 단계의 중립 디자인이 그대로 한동안 유지될 수 있으므로 "담백하지만 완성도 있게" 만든다.

## 11. 2~3단계 상세 계획 (합의 기록)

> 0~1단계(작동하는 카탈로그)는 완료. 아래는 다음 단계 **설계 합의**를 기록한 것이며,
> 실제 구현은 각 단계에 진입할 때 묶음으로 쪼개서 진행한다(0~1단계와 동일한 방식).
> **1·2·3번 기능은 모두 "로그인(Auth) + profiles 테이블"이라는 같은 토대 위에 올라간다.**

### 11.1 핵심 원칙

- **권한은 처음부터 거창하게 잡지 않는다(YAGNI).** role 한 값으로 시작하고, 매니저/검수자 등급은 실제로 필요해질 때 추가.
- **권한 체크는 UI가 아니라 RLS(DB 정책)에 둔다.** 버튼 숨김은 보안이 아니다.
- 외부 URL(앱 URL, 홍보 링크)은 전부 **새 탭 + `rel="noopener noreferrer"`**, 우리 서버에서 자동 fetch 하지 않는다.

### 11.2 2단계 — 인증 토대 (Auth + 계정)

**로그인**
- **구글 로그인(OAuth)을 기본**으로 한다. 강원이 구글 워크스페이스(도메인 `gw1.kr`)를 쓰므로 교사 가입 허들이 가장 낮다.
- (후속 여지) 이메일/비번 등 추가 수단은 필요해지면.

**profiles 테이블** (id = auth 사용자 id)
- `nickname` (닉네임)
- `avatar_url` (프로필 이미지 — Supabase Storage 업로드)
- `bio` (한 줄 소개)
- `blog_url` (네이버 블로그), `instagram_url` (인스타), `website_url` (개인 사이트) — **자기 홍보 링크, 전부 선택**
- `is_teacher_verified` (교사인증 여부)
- `role` (`member` | `teacher` | `admin` — 시작은 이 3개)
- `created_at`

**마이페이지**
- 프로필 편집(닉네임·이미지·소개·홍보 링크)
- **내 활동 모음** — 내가 등록한 앱(`owner_id = 나`), 내가 쓴 글·댓글 등을 모아 조회

**프로필 미리보기 모달 (재사용 컴포넌트)**
- 어디서나(댓글·글·앱 개발자 이름 등) **사용자명/아바타 클릭 → 모달**이 뜬다.
- 뒷배경 **흐림(backdrop blur) + 딤**. shadcn **Dialog**(토큰 기반)로 구현, ESC·포커스 트랩 기본 제공.
- 내용: 아바타, 닉네임, **교사인증 배지**, 소개, **홍보 링크**, "프로필 전체 보기".
- 트리거를 한 컴포넌트로 통일해 한 곳만 고치면 전체 반영되게.

**교사인증 (하이브리드 = 자동 + 수동)**

교사인증센터(`/verify`, 현재 ComingSoon → 실제 페이지로 채움)에서 처리한다.
**교사인증이 앱 등록의 유일한 문지기**이므로(아래 11.3) 이 절차의 신뢰성이 중요.

- **자동**: 로그인 구글 계정 이메일이 **`@gw1.kr`로 끝나면 자동으로 `is_teacher_verified = true`**. 신청 불필요.
- **수동(예외)**: 그 외 계정은 인증센터에서 **신청**한다(아래).
- ⚠️ 확인 필요: `@gw1.kr`이 교직원 전용인지(학생·행정 포함 여부). 과발급 우려가 있으면 자동인증 후에도 **관리자가 인증을 회수(내림)** 할 수 있게 둔다.

**수동 인증 신청 흐름**
1. (사용자) 인증센터 진입 → 절차 안내 읽기 → 로그인 상태에서 신청 폼 작성
   - **제출 정보**: 개인정보를 가린 **재직/경력증명서**(파일 업로드) + **근무 지역** + **학교**
2. 제출 → 화면에 **"심사 중 (1~3일 이내 처리)"** 상태 표시
3. (관리자) 대기 목록에서 서류 확인 → **승인 / 반려(사유)**
   - 승인 시: `is_teacher_verified = true`
   - **처리(승인/반려) 완료 즉시 업로드 서류 자동 삭제** — 민감정보 최소 보관. 결과값만 남기고 원본은 지움.
4. (사용자) **인증 후 최초 로그인 접속 시 "축하 + 안내" 팝업**으로 알림(이메일 발송 아님).
   - 팝업은 **한 번만** 노출(노출 여부 플래그로 관리).
   - 반려 시: 인증센터에서 사유 표시 + **재신청** 가능.

**페이지 상태별 표시 (`/verify`)**
- 비로그인 → "로그인 후 신청"
- 인증교사 → "이미 인증됨" 배지
- 미신청 → 신청 폼
- 심사중 → "심사 중 (1~3일 이내)"
- 반려됨 → 사유 + 재신청

**서류 보관 — 민감정보 특별 취급 🔒**
- **비공개(private) Storage 버킷**. 공개 버킷 금지(가렸어도 재직증명서는 개인정보).
- 접근은 **본인 + 관리자만**(RLS/서명 URL). 처리 완료 시 자동 삭제(위 3).

**verification_requests 테이블 (신청 이력 분리)**
- `id`, `user_id`, `document_path`(Storage 경로), `region`(근무 지역), `school`(학교),
  `status`(`pending` | `approved` | `rejected`), `reject_reason`, `reviewed_by`, `reviewed_at`, `created_at`
- profiles에 직접 안 넣고 별도 테이블로 → 이력·반려·재신청 처리가 깔끔. 승인 시 `profiles.is_teacher_verified` 갱신.

**관리자 검토**
- **초기: Supabase Studio로 충분**(대기 목록·서류 확인·승인 토글) → 만들 것 거의 없음, 운영 바로 시작.
- 신청이 많아지면 **`/admin`** 에 대기 목록 + 서류 미리보기 + 승인/반려 버튼.

**개발 순서 (2단계 안에서)**
1. Auth(구글) + profiles  ← 토대
2. 비공개 Storage 버킷 + verification_requests + RLS
3. `/verify` 페이지: 안내 + 신청 폼(서류·지역·학교) + 상태 표시
4. 자동 인증(@gw1.kr 도메인 체크) 결합
5. 인증 후 최초 로그인 축하 팝업
6. 관리자 검토: Studio로 시작 → 나중에 `/admin`

#### 11.2.E 묶음 E — 쪽지(1:1 메시지) *(합의 기록)*

> 2단계 토대(Auth + profiles) 위에 얹는 첫 상호작용 기능. **MVP만.**

**발신 권한**
- **로그인한 모든 사용자**가 보낼 수 있다(인증교사 한정 아님).
- ⚠️ 발신을 전체 개방했으므로, **공개 확대(이메일 인증 ON) 전에 차단(block)·신고(report)를 반드시 추가**한다.
  이번 묶음은 구조만 차단/신고를 나중에 `blocks` 테이블만 얹으면 되게 설계하고, 동작은 만들지 않는다.

**범위 (In Scope)**
- 받은함 / 보낸함 목록, 쪽지 읽기, 보내기, 안 읽음 표시, 쪽지 삭제(소프트).

**범위 밖 (후속)**
- 차단·신고·실시간 알림·답장 스레드.

**스키마 `messages`**
- `id`, `sender_id`, `recipient_id`, `body`(1~2000자), `created_at`,
  `read_at`(null = 안 읽음), `sender_deleted_at`, `recipient_deleted_at`
  (소프트 삭제 — 한쪽이 지워도 상대 사본은 유지).
- 제약: `sender_id <> recipient_id`(자기 자신에게 금지).

**RLS(보안 핵심)**
- **조회**: 내가 sender 이거나 recipient 인 행만. 또한 내 쪽에서 소프트 삭제한 행은 안 보인다.
- **발신**: `sender_id = auth.uid()` 강제(발신자 위조 차단).
- **수정**: 본문·상대는 불변. read_at·삭제표시만 바뀐다.
  → 한 행에 sender·recipient 두 당사자가 **서로 다른 컬럼**만 만질 수 있어야 하므로,
    profiles 의 단순 컬럼 GRANT 패턴으로는 당사자 구분이 안 된다.
    따라서 수정은 **`security definer` RPC 함수**(`mark_message_read`, `delete_message`)로만 한다.
    직접 UPDATE 는 회수(revoke)한다. (본문 불변·상대 불변을 확실히 보장)

**UI — 3조각으로 쪼갠다**
- **E-1**: 스키마 + RLS + RPC SQL(사용자가 적용) + 데이터 접근 일원화 `src/lib/messages.ts`
  (`getInbox`/`getSent`/`getUnreadCount`/`sendMessage`/`markRead`/`deleteMessage`).
- **E-2**: 마이페이지에 쪽지함(받은함/보낸함 탭 · 목록 · 읽기 · 안 읽음 뱃지 · 삭제).
- **E-3**: 프로필 미리보기 모달(ProfilePreview)에 "쪽지 보내기" 작성 폼(본인에겐 안 보임).

### 11.3 3단계 — 앱 등록

**흐름: 사전승인 없음**
- **인증교사**(`is_teacher_verified = true`)만 등록 가능.
- 등록 폼(제목 / 앱 URL / 썸네일 이미지 업로드 / 소개 / 카테고리) 제출 → **즉시 공개**. 검수 대기 큐·승인 화면 없음.

**apps 테이블 확장** (현재: id·title·app_url·thumbnail_url·author_name·description·category_ids·view/like/bookmark_count·created_at)
- `owner_id` (등록자 → profiles 참조)
- `status` (`published` | `hidden`) — **승인용이 아니라 사후에 내릴(숨김) 용도.** 삭제 대신 숨김이 안전.
- 썸네일은 **Supabase Storage** 버킷에 업로드.
- **쓰기 RLS 추가**: 인증교사는 본인 앱 INSERT/UPDATE, 공개 목록은 `status='published'`만, 관리자는 전체.

**개발자 홍보 연결**
- 앱 상세의 **개발자 이름 옆에 등록자의 홍보 링크(블로그·인스타·사이트) 노출** → 본 사람이 바로 그 교사 채널로. (profiles와 연결, 아이콘 링크, 새 탭)

### 11.4 이후 단계 (후속)

- 좋아요/담기/댓글 **실제 동작**(누적·토글), 조회수 누적
- **신고(report)** 버튼 → "사전승인 없음 + 사후 검수" 구조 완성
- **`/admin` 페이지**: 앱 숨김·삭제, 교사인증 수동 승인/회수, 신고 처리. (운영자가 늘거나 비개발자에게 맡길 때)
- 매니저/검수자 등 **중간 권한 등급**은 손이 부족해질 때 추가
- 과목 **하위 분류**(예: 국어 → 선긋기·자모음·낱말·받아쓰기 …): config의 Category에 `parentId` 추가, 과목 페이지 안에서 하위분류 칩으로 필터. 상위는 랜딩 그리드에, 하위는 `hideFromGrid`로 페이지 내 필터로만(= 현재 "업무자동화" 패턴 그대로). **앱이 충분히 쌓인 뒤 도입.**
